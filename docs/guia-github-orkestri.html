Conceitos Básicos
O que é Git e GitHub, sem complicar
📦 Git (local)
É um sistema que guarda o histórico de todas as mudanças no seu código. Fica instalado no seu computador. Pense como um "Ctrl+Z infinito" que sabe exatamente o que mudou, quando, e quem mudou.
☁️ GitHub (nuvem)
É onde o código fica armazenado online. Funciona como um Google Drive para código — todos da equipe conseguem baixar, enviar e ver as mudanças de forma organizada.

Seu Fluxo do Dia a Dia
Como subir atualizações quando você mesmo mexe no sistema
Antes de tudo — faça isso uma vez só (configuração do token) Para não precisar digitar senha toda vez, rode este comando no terminal do servidor ou na sua máquina onde o projeto está:
# Salva o token para não pedir toda vez (rode uma vez só) git config --global credential.helper store # Depois do primeiro push com token, ele fica salvo automaticamente
Passo a passo para subir uma atualização:
1
Abra o terminal na pasta do projeto
No Windows: abra o PowerShell ou CMD, navegue até a pasta c:\orkestri-clean
2
Veja o que mudou
git status modified: frontend/src/app/dashboard/chamados/page.tsx modified: backend/src/modules/chamados/chamados.module.ts
3
Selecione os arquivos alterados
# Selecionar tudo de uma vez (mais comum) git add . # OU selecionar arquivo específico git add frontend/src/app/dashboard/chamados/page.tsx
4
Crie um commit com descrição do que foi feito
# A mensagem descreve O QUE mudou e POR QUÊ git commit -m "fix: corrige filtro de chamados por data" git commit -m "feat: adiciona campo de prioridade no chamado" git commit -m "update: ajusta layout da página de clientes"
5
Envie para o GitHub
git push To https://github.com/orkiestriadm/orkestri-clean.git fa660e4..3a8b2c1 main -> main
Resumo em 3 linhas (copie e cole sempre que precisar) git add .                           
git commit -m "descrição do que mudou"
git push

Fluxo para Desenvolvedores
Como a equipe deve trabalhar — branches e Pull Requests
Regra de ouro — ninguém commita direto na branch main A branch main é o código de produção. Todo dev trabalha em uma branch separada e só une via Pull Request aprovado por você.
Fluxo completo do desenvolvedor:
main
(produção)
→
nova branch
feature/xxx
→
commits
diários
→
Pull
Request
→
aprovação
→ main
1
Baixar a versão mais atual antes de começar
git pull origin main
2
Criar uma branch com nome descritivo
# Padrão: feature/nome, fix/nome, update/nome git checkout -b feature/relatorio-sla git checkout -b fix/erro-login-portal git checkout -b update/layout-chamados
3
Desenvolver e commitar com frequência
git add . git commit -m "feat: adiciona gráfico de SLA por setor"
4
Enviar a branch para o GitHub
git push origin feature/relatorio-sla
5
Abrir um Pull Request no GitHub
Acesse github.com/orkiestriadm/orkestri-clean → aparecerá um botão "Compare & pull request" → clique nele → descreva o que foi feito → clique em "Create pull request".
6
Você revisa e aprova (ou pede ajustes)
No GitHub, você recebe uma notificação. Pode ver exatamente o que mudou linha por linha. Se estiver OK, clique em "Merge pull request" para unir ao main.

Regras e Boas Práticas
Para manter o projeto organizado e seguro
🔒 Segurança — o que nunca vai para o GitHub
✗ Arquivo .env (senhas do banco, chaves JWT, token do WhatsApp)
✗ Arquivos de backup (.sql, .dump)
✗ Logs do sistema
✗ node_modules (pasta de dependências)

Esses itens já estão bloqueados no arquivo .gitignore — mas fique atento.
📝 Commite com frequência, não com raridade
Prefira muitos commits pequenos do que um commit gigante no final do dia. Commits pequenos são mais fáceis de revisar e de desfazer se algo der errado.
🔀 Uma tarefa = uma branch
Cada desenvolvedor deve ter uma branch por funcionalidade que está desenvolvendo. Não misture tarefas diferentes na mesma branch.
📣 Pull Request é obrigatório para unir ao main
Nenhum dev deve rodar git push origin main diretamente. Sempre via PR revisado. Configure isso em: Settings → Branches → Add rule → require PR before merging.
Dica: proteja a branch main no GitHub Settings → Branches → Add branch protection rule → Branch name: main → marque "Require a pull request before merging". Isso impede que alguém empurre código direto sem revisão.
